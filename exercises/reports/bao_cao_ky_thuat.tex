\chapter*{Tóm tắt điều hành (Executive Summary)}
\addcontentsline{toc}{chapter}{Tóm tắt điều hành (Executive Summary)}

\begin{infobox}
Large Language Model (LLM) đang thay đổi cách con người tương tác với công nghệ. Tuy nhiên, việc đưa LLM từ phòng thí nghiệm ra sản phẩm đối mặt với rào cản lớn: \textbf{chi phí và tài nguyên tính toán}. Một mô hình 7B parameters ở FP16 yêu cầu tối thiểu 14GB VRAM, trong khi GPU consumer phổ biến nhất (T4) chỉ có 16GB.

Báo cáo này trình bày hành trình nghiên cứu và triển khai các kỹ thuật tối ưu hóa LLM trên hạ tầng hạn chế, với \textbf{câu hỏi nghiên cứu xuyên suốt}: \textit{Làm thế nào để phục vụ mô hình ngôn ngữ 7B parameters trên GPU NVIDIA T4 (16GB VRAM) với hiệu năng chấp nhận được?}

\textbf{Kết quả đạt được:}
\begin{itemize}
  \item Giảm \textbf{75\%} dung lượng mô hình (16GB $\rightarrow$ 4GB) bằng INT4 quantization
  \item Tăng \textbf{300\%} throughput (50 $\rightarrow$ 200+ tokens/s) bằng PagedAttention
  \item Giảm \textbf{75\%} latency (200ms $\rightarrow$ 50ms) bằng Redis caching
  \item Deploy production-ready với Docker, cold start $<$60s
  \item Tiết kiệm \textbf{83\%} chi phí (\$2,160 $\rightarrow$ \$360/tháng)
\end{itemize}
\end{infobox}

\section{Bối cảnh và động lực nghiên cứu}

\subsection{Sự bùng nổ của Large Language Model}

Large Language Model (LLM) đang thay đổi cách con người tương tác với công nghệ. Từ ChatGPT của OpenAI đến Llama của Meta, từ Mistral đến Qwen của Alibaba --- các mô hình ngôn ngữ lớn đã chứng minh khả năng vượt trội trong nhiều tác vụ: sinh văn bản, dịch thuật, tóm tắt tài liệu, sinh mã nguồn, và trả lời câu hỏi phức tạp.

\section{Kiến trúc Transformer và LLM}

\subsection{Cơ chế Attention}

Transformer là kiến trúc nền tảng cho hầu hết các LLM hiện đại (GPT, Llama, Mistral). Cơ chế cốt lõi là \textbf{Self-Attention} --- cho phép mô hình ``chú ý'' đến các token khác trong sequence.

\textbf{Công thức Attention:}
\begin{equation}
\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V
\end{equation}

Trong đó:
\begin{itemize}
  \item $Q$ (Query): Vector truy vấn --- ``tôi đang tìm thông tin gì?''
  \item $K$ (Key): Vector khóa --- ``thông tin nào có sẵn?''
  \item $V$ (Value): Vector giá trị --- ``nội dung thực tế là gì?''
  \item $d_k$: Chiều của Key vectors
\end{itemize}

\textbf{Multi-Head Attention} chạy nhiều attention song song, mỗi head học một khía cạnh ngữ cảnh khác nhau.

\subsection{Tại sao Attention là ``cổ chai'' về bộ nhớ}

Trong quá trình sinh văn bản, mỗi token cần lưu trữ Key và Value tensors --- gọi là \textbf{KV Cache}. Kích thước KV Cache:

\begin{equation}
\text{KV Cache} = 2 \times L \times H \times S \times B \times b
\end{equation}

trong đó $L$ = num\_layers, $H$ = hidden\_size, $S$ = seq\_len, $B$ = batch\_size, $b$ = bytes\_per\_element.

Ví dụ với Llama-3-8B (FP16): 32 layers $\times$ 4096 hidden\_size $\times$ 2048 seq\_len $\times$ 1 batch $\times$ 2 bytes = \textbf{1GB} chỉ cho KV Cache.

\subsection{Quy trình suy luận (Inference Pipeline)}

Quy trình suy luận gồm các bước:
\begin{enumerate}
  \item \textbf{Tokenizer:} Chuyển đổi text thành token IDs
  \item \textbf{Embedding:} Chuyển đổi token IDs thành hidden states
  \item \textbf{Transformer Blocks:} Xử lý qua $N$ layers (Self-Attention + FFN)
  \item \textbf{LM Head:} Chuyển đổi hidden states thành logits
  \item \textbf{Sampling:} Chọn token tiếp theo từ phân phối xác suất
\end{enumerate}

\subsection{GPTQ (GPT Quantization)}

\textbf{Nguyên lý:} GPTQ sử dụng Optimal Brain Quantization (OBQuant) --- dựa trên thông tin từ Hessian matrix để xác định trọng số nào quantize trước.

\textbf{Thuật toán:}
\begin{enumerate}
  \item Chạy calibration dataset qua mô hình gốc
  \item Tính Hessian matrix cho mỗi layer
  \item Quantize weights theo thứ tự tối ưu
  \item Bù lỗi cho các weights lân cận
\end{enumerate}

\begin{lstlisting}[language=Python,caption={GPTQ quantization}]
from auto_gptq import AutoGPTQForCausalLM, BaseQuantizeConfig

quantize_config = BaseQuantizeConfig(
    bits=4,
    group_size=128,
    damp_percent=0.01,
    desc_act=True
)

model = AutoGPTQForCausalLM.from_pretrained(model_name, quantize_config)
model.quantize(calibration_dataset)
\end{lstlisting}

\subsection{AWQ (Activation-aware Weight Quantization)}

\textbf{Nguyên lý:} AWQ nhận thấy rằng không phải tất cả weights đều quan trọng. Các weights có activation lớn (salient weights) ảnh hưởng nhiều đến output hơn. AWQ bảo vệ các weights này bằng mixed precision.

\begin{lstlisting}[language=Python,caption={AWQ quantization}]
from awq import AutoAWQForCausalLM

model = AutoAWQForCausalLM.from_pretrained(model_name)
quant_config = {
    "zero_point": True,
    "q_group_size": 128,
    "w_bit": 4,
    "version": "GEMM"
}
model.quantize(tokenizer, quant_config=quant_config)
\end{lstlisting}

\section{Tối ưu hóa tại tầng suy luận --- PagedAttention \& Batching}

\subsection{Vấn đề của Traditional KV Cache}

Trong phương pháp truyền thống, mỗi request được cấp phát một vùng nhớ liền mạch cho KV Cache. Điều này gây ra hai vấn đề:
\begin{itemize}
  \item \textbf{Memory Fragmentation:} Các request có độ dài khác nhau $\rightarrow$ vùng nhớ cấp phát dư không sử dụng được
  \item \textbf{Static Allocation:} Phải cấp phát trước cho độ dài tối đa $\rightarrow$ lãng phí nếu request ngắn
\end{itemize}

\subsection{PagedAttention --- Cơ chế hoạt động}

PagedAttention chia KV Cache thành các \textbf{pages} cố định (thường 16 tokens). Mỗi request chỉ được cấp phát đúng số pages cần thiết.

\textbf{Ưu điểm:}
\begin{itemize}
  \item \textbf{Zero waste:} Không memory fragmentation
  \item \textbf{KV Cache sharing:} Share prefix giữa các requests
  \item \textbf{Dynamic allocation:} Cấp phát theo nhu cầu thực tế
\end{itemize}

\subsection{Continuous Batching}

\textbf{Static Batching:} Mọi request trong batch phải đợi request dài nhất hoàn thành $\rightarrow$ GPU utilization $\sim$50\%.

\textbf{Continuous Batching:} Swap requests ngay khi hoàn thành, giữ GPU luôn bận $\rightarrow$ GPU utilization $\sim$85\%.

Redis caching giảm latency bằng cách lưu kết quả cho các queries lặp lại.

\begin{lstlisting}[language=Python,caption={Redis caching implementation}]
def get_cache_key(prompt, max_tokens, temperature):
    key_data = f"{prompt}:{max_tokens}:{temperature}"
    return hashlib.md5(key_data.encode()).hexdigest()

# Check cache trước khi gọi model
cached = redis_client.get(cache_key)
if cached:
    return json.loads(cached)  # Cache hit

# Cache miss - generate và lưu
result = generate_text(prompt, max_tokens)
redis_client.setex(cache_key, 3600, json.dumps(result))
\end{lstlisting}

\subsection{Tóm tắt đóng góp}

Qua 5 chương nghiên cứu, dự án đã đạt được:
\begin{enumerate}
  \item \textbf{Giảm 75\% dung lượng mô hình} (16GB $\rightarrow$ 4GB) bằng INT4 quantization với perplexity tăng $<$6\%
  \item \textbf{Tăng 300\% throughput} (50 $\rightarrow$ 200+ tok/s) bằng PagedAttention và Continuous Batching
  \item \textbf{Giảm 75\% latency} (200ms $\rightarrow$ 50ms) bằng Redis caching
  \item \textbf{Deploy production-ready} với Docker, cold start $<$60s
  \item \textbf{Giảm 83\% chi phí} (\$2,160 $\rightarrow$ \$360/tháng) so với A100
\end{enumerate}

\subsection{Ý nghĩa dự án}

\begin{successbox}
Dự án đã xây dựng một hệ thống LLM serving trên GPU T4, phục vụ hơn 100 người dùng đồng thời với chi phí \$0.5/giờ --- thấp hơn 16 lần so với giải pháp sử dụng H100. Dự án chứng minh rằng kiến thức về tối ưu hóa có thể bù đắp sự hạn chế về tài nguyên phần cứng.
\end{successbox}

\subsection{Khuyến nghị}

\begin{itemize}
  \item Tiếp tục nghiên cứu quantization cho Vietnamese language models
  \item Đánh giá speculative decoding trên T4
\end{itemize}

\textbf{Cho kỹ sư/developer:}
\begin{itemize}
  \item Bắt đầu với AWQ + vLLM cho production
  \item Benchmark trên hardware mục tiêu trước khi deploy
  \item Implement monitoring từ đầu
\end{itemize}

\textbf{Cho tổ chức muốn triển khai:}
\begin{itemize}

  \item Đầu tư vào caching và optimization trước khi nâng cấp GPU
\end{itemize}
