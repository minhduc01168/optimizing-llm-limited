Buổi 8: Đóng gói Docker & GPU Orchestration

Đóng gói Docker và Chiến lược Triển khai GPU cho LLM Production

  • Mục tiêu: Xây dựng môi trường triển khai nhất quán, bảo mật và hiệu năng cao.
  • Trọng tâm: Làm chủ NVIDIA Container Toolkit, tối ưu hóa kích thước image và cấu hình tài nguyên GPU.

Tại sao phải sử dụng Docker cho LLM?

  • Nhất quán môi trường: Loại bỏ xung đột phiên bản CUDA, thư viện Python và Driver giữa các môi trường dev, staging và production.
  • Khả năng tái lập: Định nghĩa các phụ thuộc một lần và đảm bảo chúng hoạt động giống hệt nhau ở mọi nơi.
  • Tách biệt tài nguyên: Quản lý tài nguyên GPU và bộ nhớ độc lập cho từng mô hình.

NVIDIA Container Toolkit - Cầu nối GPU

  • Chức năng: Cho phép môi trường Docker truy cập trực tiếp vào phần cứng GPU và trình điều khiển (Driver) của máy chủ vật lý.
  • Cơ chế: Ánh xạ các node thiết bị NVIDIA từ host vào bên trong container.
  • Lưu ý quan trọng: Phiên bản CUDA bên trong image (ví dụ: 12.1) phải tương thích với phiên bản Driver NVIDIA trên máy chủ (ví dụ: 525.60+).

Lựa chọn Base Image tối ưu

Mẹo: Luôn ghim (pin) tag phiên bản cụ thể thay vì dùng :latest để tránh lỗi bất ngờ.

Trường hợp sử dụng Base Image khuyến nghị
Triển khai vLLM vllm/vllm-openai:v0.6.0
Triển khai TGI ghcr.io/huggingface/text-generation-inference:2.3.0
Tùy chỉnh (Custom) nvidia/cuda:12.1.0-cudnn8-devel-ubuntu22.04
Inference nhẹ nvidia/cuda:12.1.0-runtime-ubuntu22.04

Tối ưu kích thước Image với Multi-Stage Builds

  • Vấn đề: Image LLM thường rất lớn (10-20GB) do chứa các header phát triển CUDA và thư viện build.
  • Giải pháp: Sử dụng Multi-stage build để tách biệt giai đoạn xây dựng và giai đoạn chạy.
  • Giai đoạn 1 (Builder): Cài đặt công cụ build, biên dịch kernel CUDA và cài đặt thư viện.
  • Giai đoạn 2 (Runtime): Chỉ copy các file thực thi và thư viện cần thiết sang một image runtime nhỏ gọn hơn.
  • Kết quả: Có thể giảm dung lượng image xuống hơn 50%.

Cấu hình Shared Memory (--shm-size)

  • Tại sao cần? Các framework như vLLM và PyTorch sử dụng bộ nhớ chia sẻ (shared memory) để giao tiếp giữa các tiến trình tensor-parallel.
  • Rủi ro: Giá trị mặc định của Docker (64MB) sẽ gây crash hệ thống ngay khi bắt đầu xử lý đồng thời.
  • Khuyến nghị: Thiết lập ít nhất 16GB cho cấu hình đơn GPU và 32GB cho đa GPU.

GPU Passthrough và Memory Utilization

  • GPU Passthrough: Sử dụng flag --gpus all hoặc định danh thiết bị cụ thể để cấp quyền truy cập phần cứng cho container.
  • --gpu-memory-utilization: Tham số kiểm soát tỉ lệ VRAM mà engine sẽ chiếm dụng.
  • Khuyến nghị: Đặt ở mức 0.90 để dành lại 10% bộ đệm cho các overhead của CUDA và framework, tránh lỗi Out-of-Memory (OOM) khi có lưu lượng đột biến.

Quản lý bộ nhớ Persistent (Volume Mounting)

  • Quy tắc vàng: Không bao giờ nén (bake) trọng số mô hình trực tiếp vào Docker image.
  • Giải pháp: Gắn kết (mount) thư mục cache (ví dụ: ~/.cache/huggingface) dưới dạng Volume.
  • Lợi ích:
    • Tránh tải lại mô hình (vài chục GB) mỗi khi khởi động lại container.
    • Dễ dàng cập nhật phiên bản mô hình mà không cần build lại image.

Orchestration đa GPU: Tensor Parallelism

  • Khi nào cần? Khi mô hình quá lớn không vừa một GPU đơn lẻ (ví dụ: Llama-3-70B cần 140GB ở FP16) hoặc cần giảm độ trễ (latency).
  • Tensor Parallelism (TP): Chia nhỏ các lớp của mô hình để xử lý song song trên nhiều GPU cùng lúc.
  • Cấu hình: Sử dụng tham số --tensor-parallel-size tương ứng với số lượng GPU được cấp phát.

Yêu cầu kỹ thuật cho Đa GPU (Multi-GPU)

  • --ipc=host: Yêu cầu bắt buộc để kích hoạt giao tiếp NCCL (NVIDIA Collective Communications Library) giữa các GPU.
  • Băng thông: Hiệu năng TP phụ thuộc lớn vào tốc độ kết nối nội bộ giữa các GPU (như NVLink hoặc InfiniBand).
  • Đồng nhất: Các GPU trong một cụm nên có cùng kiến trúc và dung lượng VRAM để tránh hiện tượng nghẽn cổ chai.

Thiết lập Health Checks cho LLM Container

  • Thách thức: LLM mất nhiều thời gian (30-180 giây) để nạp trọng số từ đĩa vào VRAM trước khi sẵn sàng phục vụ.
  • Cấu hình start_period: Thiết lập khoảng thời gian chờ (ví dụ: 120-300 giây) trong Docker Compose để tránh việc Docker giết container khi nó đang nạp mô hình.
  • Lệnh kiểm tra: Thường kiểm tra endpoint /health hoặc /v1/models của engine phục vụ.

Kiến trúc Production Docker Compose

Một hệ thống triển khai tiêu chuẩn nên bao gồm:

  • Inference Engine (vLLM/TGI): Xử lý suy luận trên GPU.
  • API Gateway (FastAPI): Quản lý xác thực và hàng đợi.
  • Redis: Lưu trữ trạng thái task và hàng đợi thông điệp.
  • Prometheus: Thu thập các chỉ số (metrics) về VRAM và throughput.

Bảo mật Container trong Production

  • Non-root User: Tránh chạy container bằng quyền root để hạn chế rủi ro nếu container bị tấn công.
  • Secrets Management: Không để API Token (HuggingFace) trong biến môi trường của file compose; hãy sử dụng Docker Secrets hoặc file .env được bảo vệ.
  • Network Isolation: Chỉ công khai cổng (port) của API Gateway, giữ engine suy luận trong mạng nội bộ Docker.

Xử lý các lỗi triển khai phổ biến

  • Container OOM khi khởi động: Giảm gpu-memory-utilization hoặc bật Quantization (AWQ/GPTQ).
  • Mô hình không nạp được: Kiểm tra HuggingFace Token hoặc đường dẫn mount volume.
  • Lỗi NCCL/Hàng đa GPU: Kiểm tra flag --ipc=host và sự đồng nhất của driver giữa các GPU.
  • Inference chậm bất thường: Kiểm tra biến môi trường CUDA_VISIBLE_DEVICES xem có vô tình để trống khiến engine chạy trên CPU không.

Tổng kết và Best Practices

  • Ưu tiên sử dụng Multi-stage builds để tối ưu tốc độ triển khai.
  • Luôn cấu hình Shared Memory và Health Checks phù hợp với kích thước mô hình.
  • Sử dụng Persistent Volumes để quản lý trọng số mô hình một cách linh hoạt.
  • Giám sát các chỉ số VRAM (gpu_cache_usage_perc) để điều chỉnh quy mô hạ tầng kịp thời.

Bài tập Thực hành DevOps: Viết Docker Compose cho vLLM

Dựa trên lý thuyết đã học, hãy hoàn thiện file docker-compose.yml để triển khai mô hình LLM với GPU.

Bài 1: Cấu hình Tài nguyên và Volume

Yêu cầu: Bổ sung các thông số còn thiếu vào file cấu hình dưới đây để đảm bảo:
1. Gắn tất cả GPU có sẵn vào container.
2. Set bộ nhớ chia sẻ (shared memory) lên 16GB.
3. Mount thư mục ./model_cache trên máy host vào /root/.cache/huggingface trong container.

version: '3.8'

services:
  vllm-server:
    image: vllm/vllm-openai:v0.6.0
    ports:
      - "8000:8000"
    environment:
      - HUGGING_FACE_HUB_TOKEN=${HF_TOKEN}
    command: --model meta-llama/Llama-3-8B-Instruct --gpu-memory-utilization 0.90
    
    # --- ĐÁP ÁN BẮT ĐẦU TỪ ĐÂY ---
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: all
              capabilities: [gpu]
              
    shm_size: '16gb'
    
    volumes:
      - ./model_cache:/root/.cache/huggingface
